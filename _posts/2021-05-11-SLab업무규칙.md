## 사전준비

### 용량제안
1. AP서버
 - 제안 단계에서 오브젠 제안팀은 서버의 CPU / Memory 용량을 산정 합니다.

1. Database서버
 - 제안 단계에서 오브젠 제안팀은 DB서버의 Storage / CPU / Memory 용량을 산정 합니다.

### 인프라설계
 - 프로젝트에서 고객사 IT와 준비 / 환경을 협의 합니다.
 - 프로젝트에서 고객사 IT와 설치 일정을 협의 합니다.
 - 프로젝트는 설치 일정을 SL팀에 통지 합니다.

1. Internal HDD 구성
 - SW영역은 10G 이상을 권장합니다.
 - Log영역은 100G이상을 권장합니다.
 - Temp영역은 문서를 저장하는 용도입니다. 200G이상을 권장합니다. 서버가 이중화 된 경우에는 NAS 영역이 필요 합니다.

1. Database Storage 구성
 - SAN Disk 연결은 고객사 IT가 설치합니다.
 - 볼륨구성은 프로젝트에서 제안된 (필요한) Data영역의 크기를 고객사 IT에 제출합니다.  이를 근거로 고객사 DBA가 구성을 정의합니다.

1. IP 설정 및 L4이중화
 - 물리 / 가상 IP는 고객사 IT가 정의 합니다.
 - 서버가 이중화 된 경우 L4연결이 필요합니다. 이때 연결 규칙은 Hash 사용이 필수 입니다. (OLAP, 예의 사용을 위해서는 동일 PC에서 항상 동일 서버로 연결 되어야합니다.)

1. 방화벽 설정
 - SL팀은 솔류션 운영울 위한 Port 정보를 제공한다.
 - 프로젝트는 고객사 IT에게 서버 방화벽, Network 설정, L4 설정을 요청한다.

### 데모PC설치
 - 프로젝트에서 관리부에 PC를 요청합니다.
 - 프로젝트에서 설치를 수행합니다.
(표준 제품은 SL팀에서 최신 Meta를 제공합니다. 고객사에 데모한 버전이 필요하면 컨설팅팀에서 Meta를 제공 합니다.

## 설치

### 인프라준비
1. OS설치
 - OS설치는 고객사 IT의 지시로 서버업체에서 설치 합니다. (오브젠에서 서버 납품 시 에는 영업팀에서 서버도입 비용으로 협력사에 설치를 요청 합니다.)

1. JDK설치
 - JDK 1.8 (build 100이상)이 필요 합니다.
 - 고객사 IT가 설치 합니다.  
(오브젠에서 설치하는 경우 협력사에 프로젝트 비용으로 요청한다. 이 경우도 고객사 IT가 인수 합니다.)

1. WEB/WAS설치
 - 고객사 IT가 WAS 업체에게 설치를 지시합니다.
 - 고객사 IT가 SSL 적용 합니다. (인증서는 고객사 IT가 구매 합니다.)
(오브젠에서 설치하는 경우 협력사에 프로젝트 비용으로 요청한다. 이 경우도 고객사 IT가 인수 합니다.)

1. Database 설치
 - 고객사 IT가 WAS 업체에게 설치를 지시합니다.
(오브젠에서 설치하는 경우 협력사에 프로젝트 비용으로 요청한다. 이 경우도 고객사 IT가 인수 합니다.)

### Obzen제품 설치
1. 설치 전 준비사항
 - 프로젝트에서 "사전준비-인프라설계" 단계에서 정의한 설치 일정에 맞추어 고객사 IT에게 설치환경 및 사전작업(방화벽 신청 등)을 2주 전에 요청합니다.
<<별첨 - 설치사전준비사항.xls>> - URL
 - 고객사IT는 설치 예정일 전에 프로젝트에 설치환경과 사전준비 완료를 통보합니다.

1. 제품 설치 (개발 환경)
 - SL팀은 설치 일정에 설치 매체(MetaKit)을 제공 합니다.
(Obzen Framework - Server 모듈과 각 제품별 Meta Data, Web 메뉴 등) 
 - 프로젝트에서는 설치파일의 반입 절차를 확인하여 고객사IT에게 신청합니다.
 - 프로젝트는 개발서버를 Default Config로 설치 합니다.
(Sample마트는 설치 대상이 아닙니다. 프로젝트는 필요 시 시전에 SL팀에 요청합니다.)

1. 제품 설치 (운영 환경)
 - 프로젝트는 사용자 수 (등록 / 로그인 / 동시) / 캠페인 수 (일별)를 Config 생성기에 입력하여 Config파일을 생성 한다. 
 - 프로젝트는 개발서버의 개발 결과 (Obzen Framework, Meta, Web메뉴 등)를 운영서버로 이관 합니다.
 - 프로젝트는 설치를 수행합니다. 
 - 프로젝트는 설치확인서의 고객 승인을 득한 후 오브젠 영업대표에게 사본을 제출합니다.

1. 마케팅 관리 (JDW11)
 - 프로젝트는 UI를 통해 입력한 Content (코드 등록/변경, 제외조건, 성공조건 보고서 등)와 인터페이스(감지채널, 발송채널 등)를 운영서버에 등록합니다.
 - 설치 이후 변경 사항은 프로젝트에서 개별 이관 합니다.
 - 프로젝트에서 업무요건에 따라 개발(감지채널 설계, 실시간캠페인 운영관리 요소 등)을 운영서버에 입력 / 개발 합니다.
(마트는 설치 대상이 아닙니다. 프로젝트에서 생성합니다.)

1. Data Planet
 - 프로젝트는 정기 보고서 발송을 위한 SMTP(이메일) 서버 연결을 개발 합니다.
(정기 보고서는 메타 이행시 등록 됩니다.)


1. 설문
  - 프로젝트는 업무요건에 의해 개발된 Web Source (html, css, jsp 등)를 운영서버에 배포합니다.

1. D3F
 - 프로젝트는 업무요건에 의해 개발된 설정(D3F 운영관리, DP연계 등)을 운영서버에 등록합니다.

1. Web 화면
 - 프로젝트는 업무요건에 의해 개발된 HTML을 운영서버에 등록합니다.

##공통기능

1. 고객사 단말 시스템에 임베디드
 - 고객사 단말 시스템에 솔류션 화면을 임베디드 하는 경우 SL팀은 frameloader.html을 프로젝트에 제공합니다. (Web메뉴에 포함 제공 합니다.)
 - SL팀은 framelaoder.html의 호출방식을 프로젝트에 가이드 합니다.
 - 프로젝트는 단말 메뉴로 frameloader.html을 사용한 url을 작성하여 등록합니다.

1. SSO 적용
 - 고객사 포탈에서 Link로 연결하는 경우 , 고객사 SSO와 연동 합니다.
 - 프로젝트에서 고객사 IT로부터 적용가이드를 받아 SL팀에 제출합니다.
 - SL팀이 가이드 확인 후 방문하여 적용 합니다.

1. 사용자 계정 동기화
 - 고객사 인적자원(HR) 테이블에서 캠페인 관리 계정과 연동 합니다.
 - 프로젝트에서 사용자 / 조직 목록을 조회하는 SQL을 작성하여 SyncUser를 설정 합니다.
 - 프로젝트는 Agent 스켸쥴에 SyncUser를 등록 합니다.
(삭제 재등록 방식은 비밀번호 관리가 않 될 수 있습니다.)

1. 형상관리
 - 고객사 형상관리를 통해 개발대상(Web, Frame과 TM)을 운영서버에 배포합니다. (Cube / 보고서 등 UI를 사용하여 입력하는 내용은 대상으로 하지 않습니다.)
 - 프로젝트에서 고객사 IT와 형상관리의 소스저장 폴더와 파일종류 등을 확정합니다. (Export한 Web 화면 관리여부 등) 
 - 설치 시 복사하는 Obzen Framework과 Web 자원(HTML, JS, CSS)는 제품 패치로 제공 됩니다. (형상관리 대상 아님)
 - SL팀에서 Frame과 TM의 운영 서버에 적용하는 Shell을 제공 및 테스트 지원 합니다. (Web자원은 고객사 형상관리가 표준 방법으로 이관 합니다.)
 - 고객사 IT는 형상관리에 SL팀이 제공한 Shell을 호출하도록 개발 / 테스트를 합니다. 
 - 프로젝트에서 고객사 SR시스템에 소스 등록과 운영 이관을 수행 합니다.

1. Online 서비스 개발
 - 고객사 타 시스템에서 고객별 캠페인 성과 / 대상 캠페인 정보를 OnLine형태로 질의 하는 경우, RealTimeServer의 TM 호출 기능을 이용해 서비스를 개발한다.
 - SL팀은 서버 Jar / Config 와 전문 수신 Sample (http / json 방식) 코드를 제공 합니다.
 - 고객사 IT는 표준 전문 포멧과 필요시 EAI 통신 Jar(Java 1.8용) 파일과 적용 가이드를 제공합니다.
 - 프로젝트에서 전문 수신 코드를 개발 / 테스트 합니다.
 - 프로젝트에서 업무 처리용 서비스(TM)를 개발 / 테스트 합니다.
<<별첨 – Online서비스가이드.ppt>> 이준희C 작성
1. Cube 개발
 - 프로젝트에서 고객사의 업무를 분석하고, Cube를 작성합니다.
 - 프로젝트는 SL팀이 제공한 Cube 개발 가이드를 참고하여 오브젠 OLAP제품요으로 최적화를 수행합니다.
<<별첨 Cube 최적화 가이드.ppt>> 최형석I 작성

## 제품 커스텀
 - 프로젝트는 제품 커스텀 금지입니다.
 - 프로젝트는 고객요청 사항이 제품 커스텀을 필요로하는 경우, 요구사항 수집 단계에서 연구소와 커스텀 여부를 협의 합니다. (계약 단계에서 커스텀 비용이 있는 경우만 해당 합니다.)
 - 연구소 반려 시 요구사항을 반려 합니다. (꼭 필요 한경우 제품 로드맵 협의를 연수소에 요청 합니다.)

1. 예외 범위
 - 제품이 아닌 부분은 프로젝트가 판단하여 개발 합니다. (인터페이스, 감지, 보고서 등)
 - 제품 UI를 통해 입력하는 컨텐츠는 프로젝트가 판단하여 개발합니다. (제외, 성공조건, Cube 등)
 - 마트, ETL등은 프로젝트가 판단하여 개발 합니다. (D3F 포함)
 - 예외 범위는 SL팀의 하자 / 장애조치 대상이 아닙니다. PM을 통해 프로젝트에서 조치 합니다.

## 성능테스트

### 진행절차
 - 프로젝트는 개발단계 진입 직후 성능테스트 목표를 고객사 IT와 합의 합니다.
 - 프로젝트는 성능 테스트 시나리오를 준비 합니다.
 - 프로젝트는 성능 테스트를 위한 스크립트를 준비 합니다. 
 - 프로젝트는 성능 테스트를 수행 합니다. (부하 발생기는 고객사 IT가  준비 합니다.)
 - 프로젝트는 튜닝을 통해 목표 성능을 달성 합니다.

1. 운영AP Config
 - 프로젝트는 사용자수 / 월별 캠페인 수 / 감지 규모 등을 Config 생성기에 입력하여 운영서버 설치 시 운영 Config를 최적화 합니다.

1. 캠페인 관리 성능 테스트 수행
 - 고객사 IT와 프로젝트는 성능 테스트 도구를 준비 합니다. (LoadRunner or JMeter)
 - 프로젝트는 성능테스트 계획 작성과 수행 시나리오를 준비합니다.
 - 성능테스트 도구에 시나리오에 해당하는 스크립트를 Web 화면에서 발췌하여 등록 합니다.
 - 프로젝트는 성능테스트를 수행 합니다.  고객사 IT는 운영서버 자원 (CPU / Memory) 사용을 모니터링 합니다.
 - 프로젝트는 병목 부분을 식별하고, SQL 최적화를 통해 성능을 개선 합니다. (병목 중 엔진부분은 SL팀에 전달 합니다.)
 - 프로젝트는 성능테스트 결과를 분석 / 보고서를 작성하여 고객사 IT에 제출 합니다.
<<별첨 성능테스트 가이드.ppt>>

1. 실시간 캠페인 성능 테스트 수행
 - 프로젝트는 성능 테스트를 위한 대량의 감지 데이터를 준비 합니다. (DB Pooling / EAI 전송)
 - 프로젝트는 성능테스트 계획 작성과 수행 시나리오를 준비 합니다.  (캠페인 등록을 수행 합니다.)
 - 프로젝트는 성능테스트를 수행 합니다.  고객사 IT는 운영서버 자원 (CPU / Memory) 사용을 모니터링 합니다.
 - 프로젝트는 큐 / 작업 스레드 최적화를 통해 성능을 개선 합니다. (병목 중 엔진부분은 SL팀에 전달 합니다.)
 - 프로젝트는 성능테스트 결과를 분석 / 보고서를 작성하여 고객사 IT에 제출 합니다.

## 프로젝트 안정화

1. 운영자 가이드 작성
 - 프로젝트는 운영자 메뉴얼을 작성 합니다. (개발기->운영기 소스 이관 방법, 개발/운영 서버 기동/중지 방법, SSO/SyncUser 적용 설명을 꼭 포함 합니다.)

1. 운영자 인수/인계
 - 프로젝트는 고객사 IT 운영자(들)를 식별 합니다.
(운영자가 식별 되지 않는 경우 영업대표에게 SM투입을 요청 합니다.)
 - 프로젝트는 운영자(들)에게 인수/인계를 수행 합니다.

## 프로젝트 종료

1. 운영 라이센스 발급
 - 프로젝트는 세금계산서 발급 후 운영서버의 라이센스를 SL팀에 요청합니다.
 - SL팀은 라이센스를 발급(연구소)하여 프로젝트에 전달합니다.
 - 프로젝트는 운영서버에 라이선스 파일(LockFile)을 적용 합니다. (프로젝트 인원이 철수한 경우 고객사 운영자에게 제공 합니다.)

1. SL팀에 사이트 인계
 - 프로젝트는 최종검수 및 세금계산서발급과 "프로젝트 안정화" 단계가 끝난 뒤 SL팀에 사이트 관리를 요청합니다. (고객사 운영자 식별이 필요 합니다.)
 - SL팀은 담당 인원을 식별합니다.
 - 프로젝트는 SL팀과 사이트의 구성, 유지보수 리스크 및 반려 요건 등을 인수/인계 합니다.

1. 하자 조치
 - 고객사 운영자는 SL팀에 하자를 통보하고, 조치를 요청합니다.
 - SL팀은 하자여부를 확인 한 후 하자조치를 연구소에 요청 합니다. (신규 / 개선은 반려합니다.)  
 - SL팀은 제품 패치를 고객사 운영자에게 제공합니다.
 - 제품 외 범위에서 장애가 발생한 경우 PM에게 하자조치 요청을 전달 합니다.  
   프로젝트는 고객사를 방문하여 하자 내용 파악과 조치를 수행합니다. (PM이 외주 회사인 경우 영업을 통해 요청합니다.)

1. 장애 발생
 - 장애발생 시 고객사 IT는 SL팀에 장애를 통보하고, 조치를 요청합니다.
 - SL팀은 방문 점검하여 원인을 파악하고, 이를 조치합니다.
 - 제품 외 범위에서 장애가 발생한 경우 PM에게 장애 조치 요청을 전달 합니다.  
   프로젝트는 고객사를 방문하여 장애 원인 파악과 조치를 수행합니다. (PM이 외주 회사인 경우 영업을 통해 요청합니다.)
